# Multi-Modal Evidence Review Agent

A VLM-powered damage claim verification system for HackerRank "Multi-Modal Evidence Review" challenge. Verifies car/laptop/package damage claims using images, conversation transcripts, user history, and minimum evidence requirements.

## Run & Operate

```bash
# Run the agent (from repo root, requires OPENAI_API_KEY or ANTHROPIC_API_KEY)
python code/agent.py

# With explicit paths
python code/agent.py \
  --claims dataset/test.csv \
  --history data/user_history.csv \
  --evidence-reqs data/evidence_requirements.csv \
  --output output.csv \
  --log log.txt \
  --data-root .

# Use Anthropic Claude instead of OpenAI
python code/agent.py --use-anthropic

# Quick test (first 3 claims only)
python code/agent.py --sample 3

# Evaluate predictions against ground truth
python code/evaluate.py --ground-truth data/sample_claims.csv --predictions output.csv
```

Required env: `OPENAI_API_KEY` (GPT-4o vision, needs credits) or `ANTHROPIC_API_KEY` (with `--use-anthropic`)

## Stack

- Python 3.11
- OpenAI GPT-4o vision (primary) / Anthropic Claude 3.5 Sonnet (alternative)
- openai, anthropic, tenacity, tqdm, pandas

## Where things live

- `code/agent.py` — main entry point
- `code/analyzer.py` — VLM call logic, JSON parsing, result normalization
- `code/prompts.py` — system prompt + analysis prompt templates
- `code/utils.py` — image encoding, CSV I/O, logging, prompt injection detection
- `code/evaluate.py` — evaluation workflow (compare predictions vs ground truth)
- `dataset/test.csv` — 44 test claims to predict
- `data/user_history.csv` — per-user risk history
- `data/evidence_requirements.csv` — minimum evidence rules by object type
- `data/sample_claims.csv` — labeled examples for evaluation
- `images/test/` — 82 test images
- `images/sample/` — 29 labeled sample images
- `output.csv` — agent predictions (HackerRank submission)
- `log.txt` — chat transcript (HackerRank submission)

## Architecture decisions

- Images are the primary source of truth — the VLM decision is grounded in what is visually visible, not in claim text alone
- Prompt injection is detected both at the pre-screening stage (regex patterns in `utils.py`) and by the VLM system prompt, which is instructed to flag and ignore manipulation attempts
- User history adds risk flags only — it never overrides clear visual evidence
- All 11 evidence requirements from `evidence_requirements.csv` are injected into the VLM prompt for the relevant object type
- Fallback results are returned on API failures so the CSV always has all 44 rows

## Product

For each damage claim the agent outputs:
- `evidence_standard_met` — whether images meet the minimum evidence bar
- `claim_status` — supported / contradicted / insufficient_evidence
- `issue_type` — crack, dent, scratch, broken_part, etc.
- `object_part` — specific part (rear_bumper, screen, package_corner, etc.)
- `risk_flags` — blurry_image, claim_mismatch, non_original_image, user_history_risk, manual_review_required, prompt_injection_detected
- `severity` — low / medium / high / critical / none
- `supporting_image_ids` — which images support the decision

## HackerRank submission

1. **Code zip**: `zip -r code.zip code/` (exclude virtualenvs, node_modules, data)
2. **Predictions CSV**: `output.csv` (run the agent first)
3. **Chat transcript**: `log.txt` (auto-generated by the agent)

## Gotchas

- Run `python code/agent.py` from the **repo root** (not from inside `code/`), so image paths like `images/test/case_001/img_1.jpg` resolve correctly
- OpenAI free-tier keys have `insufficient_quota` — you need a funded account or use `--use-anthropic`
- The agent writes `output.csv` incrementally — if interrupted, re-run from scratch (or add `--sample N` to test a subset first)
